sol-login component

Live demo: (no events yet — click the button or trigger an authed fetch elsewhere)

Login/logout UI plus the page-wide authenticated-fetch plumbing. A single <sol-login> on the page does three jobs: (1) lets the user log into one or more Solid pods, (2) provides authenticated fetch to every other sol-* component via the shared AuthManager, and (3) listens for the sol-auth-needed event so any fetch that returns 401 (or 403 with no active session) triggers the login flow automatically — the calling widget never has to know auth exists.

Prerequisites

To use Solid login and authenticated fetch, apps must import the Inrupt authentication library. The simplest setup is to import the prebuilt UMD that ships in dist/vendor/:

<script src="../dist/vendor/@inrupt-solid-client-authn-browser.umd.js"></script>

Usage

Basic Usage

With no attributes, the tag prompts the user for an issuer, then logs in via a full-page redirect.

<sol-login></sol-login>

Pre-configured Issuers

The issuers attribute provides quick-select buttons in the login dropdown.

<sol-login issuers="https://solidcommunity.net, https://login.inrupt.com"></sol-login>

Modes: redirect vs popup

The mode attribute picks the OIDC flow:

  • redirect (default) — classic full-page OIDC redirect. One session per page. State on the page is lost during the redirect. Fine for a dedicated login page; not recommended when auth might be triggered mid-flow by an auto-save.
  • popup — login happens in a small popup window that holds the real session; the parent talks to it through a PopupProxySession. The calling page never reloads, so an interrupted save can resume on retry. Multiple <sol-login side="…"> elements can hold independent sessions in the same tab.
<sol-login mode="popup"
           popup-callback="popup-auth-callback.html"
           issuers="https://solidcommunity.net,https://login.inrupt.com">
</sol-login>

Tagged sessions (side)

Each <sol-login> instance is keyed by its side attribute (default: "default"). In popup mode, two elements with different side values can be logged in simultaneously (e.g. podz left/right pods each holding their own session).

<sol-login mode="popup" side="left"  issuers="…"></sol-login>
<sol-login mode="popup" side="right" issuers="…"></sol-login>

Event Handling

Listen for sol-login / sol-logout to trigger app-specific actions. Both events bubble and are composed, so listening at document level catches them from any sol-login on the page including ones inside shadow roots (e.g. inside sol-pod).

document.addEventListener('sol-login',  (e) => console.log('Logged in as', e.detail.webId, 'via', e.detail.issuer));
document.addEventListener('sol-logout', (e) => console.log('Logged out:', e.detail.side));

Automatic auth via solFetch

The recommended way to fetch a Solid resource is solFetch, exported from core/auth-fetch.js:

import { solFetch } from 'solid-web-components/core/auth-fetch.js';

const r = await solFetch('https://example.pod/private/notes.ttl', {
  method: 'PATCH', headers: { 'content-type': 'text/n3' }, body: …,
});

If the response is 401 (or 403 with no active session), solFetch dispatches a sol-auth-needed event on document. A <sol-login> anywhere on the page receives the event, runs its configured login flow, and resolves the waiting promise. solFetch then retries the original request once. The calling widget never has to handle the 401 itself.

When no <sol-login> is mounted, the event is not dispatched and solFetch returns the 401 response as-is — so widgets that need to work in unauthenticated contexts (static pages, demos) still behave sensibly.

sol-auth-needed event contract

For frame authors who want to hook the login UX (e.g. a custom chrome that shows a banner during auth):

document.addEventListener('sol-auth-needed', (e) => {
  const { url, response, side, resolve, reject } = e.detail;
  // url      — the resource being fetched
  // response — the original Response (status 401 or 403)
  // side     — the authTag the request was made under (or undefined for
  //            untagged calls). When set, ONLY the <sol-login> with
  //            matching `side` handles the event; others early-return.
  //            Untagged events fall through to every listener — first
  //            to resolve wins.
  // resolve(true)  → solFetch retries the original request
  // resolve(false) → solFetch returns the original response
  // reject(err)    → solFetch throws err
});

The built-in <sol-login> listener picks the default issuer (own issuer attr → <sol-default default-issuer="…"> → first entry of the issuers list) and runs the configured login flow. Concurrent prompts are coalesced into one login attempt. A five-minute timeout caps the wait so an ignored popup doesn't hang the calling widget forever.

Pages with multiple <sol-login> chips (e.g. podz's left + right per-pod chips) get side-scoped routing: a solFetch(url, {authTag: 'left'}) only wakes the side="left" chip. Callers without an authTag still wake every chip — same as before.

Visibility model

<sol-login> is hidden by default (:host { display: none }). It surfaces only while an auth prompt is in flight — the host element gains the active attribute and the rule :host([active]) flips display back on. After success/cancel/popup-blocked the attribute clears and the element returns to invisible.

Hosts that DO want a permanent login chip can override the default with inline style, or use the visible attribute:

<sol-login visible mode="popup" issuers="…"></sol-login>

Frames that prefer to show login state somewhere else (e.g. paint another chrome chip green when logged in) listen for the sol-login / sol-logout events and keep sol-login itself hidden.

Default issuer via <sol-default>

<sol-login>'s auth-needed listener reads its default issuer in this order:

  1. The element's own issuer attribute.
  2. <sol-default default-issuer="…"> elsewhere on the page.
  3. The first entry of the issuers list.
<sol-default default-issuer="https://solidcommunity.net"></sol-default>
<sol-login mode="popup" issuers="https://solidcommunity.net,https://login.inrupt.com"></sol-login>
