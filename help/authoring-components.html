Add your component to the ecosystem

Newly implemented. All of this is shipped, including the manifest's imports block in Step 2. It hasn't been exercised end-to-end in a real browser yet, so treat the runtime details as provisional and report anything that misbehaves.

You wrote a web component. This guide takes you through making it a good citizen of a solid-web-components page, step by step — so other people can drop it in without fuss, and capabilities like editing can enhance it. Each step is optional and independent; do the ones you need.

You never import any solid-web-components source code. Everything is plain custom elements, a JSON manifest, and DOM events.

Step 1 — Make it a custom element

Give it a hyphenated tag with your own prefix (so it never collides with anyone else's), and register it:

class AcmeMap extends HTMLElement { /* … */ }
customElements.define('acme-map', AcmeMap);

That's the whole of plugging in. Optionally, if you want other parts of the page — app code, other components, even across shadow DOM — to react when something happens in yours, announce it with a bubbling, composed event named your-tag-action. Nothing in the loader or the extension mechanism needs this; it's just how components coordinate on a page when they want to:

this.dispatchEvent(new CustomEvent('acme-map-select', {
  bubbles: true, composed: true, detail: { id },
}));

Step 2 — Create a manifest (so people use it without the details)

You know the messy details of your component — where its module lives, what it offers. Write them down once, in a small JSON manifest you ship alongside your component (in your npm package, or a file people drop in). After that, nobody using your component needs to know any of it.

A manifest has two parts, both optional:

{
  "name": "acme-components",

  "imports": {
    "acme-map":   "https://cdn.acme.example/acme-map.js",
    "acme-chart": "https://cdn.acme.example/acme-chart.js"
  },

  "capabilities": {
    "acme-viz": { "modules": ["acme-map", "acme-chart"] }
  }
}
  • imports — where each of your components actually lives. This is the part that spares your users the URLs. Your entries can point anywhere (a CDN, a package path), but can't override a library the page already shares, so the page stays coherent. (Want to offer both an offline build and a CDN form? Key them by stage — see below.)
  • capabilities — only if you provide a capability others switch on with data-extend-with (see Step 4).

What the person who uses your component writes

Their entire integration is one loader tag pointing at your manifest, plus your tag. No URLs, no import map, no build:

<script src="dist/sol-loader.min.js"
        data-stage="local"
        data-manifest="acme/swc-manifest.json"
        data-bundles="acme-map"
></script>

<acme-map src="data/places.ttl"></acme-map>

The loader reads your manifest, learns where acme-map lives, and loads it. The details stayed with you, the author, where they belong.

Offer both an offline and a CDN build (optional)

Because they're on the loader, you can hand your users one more convenience for free: a deployment switch. Key your imports by stage in the same manifest…

{
  "stages": {
    "local": { "imports": { "acme-map": "vendor/acme-map.js" } },
    "cdn":   { "imports": { "acme-map": "https://cdn.acme.example/acme-map.js" } }
  }
}

…and the consumer's data-stage picks which — local for a self-hosted/offline build, cdn to pull from the internet. The one knob covers your components and the loader's own pieces. (A flat top-level imports still works if you don't care which.) See Loading the components.

You can stop here. Steps 1–2 are a complete, useful integration on their own: plenty of components use the loader purely to give people clean includes. The rest are optional extras — only if you want your component enhanced (Step 3) or want it to provide a capability (Step 4).

Step 3 — (Optional) Let capabilities enhance your component

Declare the extension points you offer with one static getter. The handy one is edit: point it at a small file listing your fields and the page's editing capability builds your component a settings form and a ⚙ gear, for free:

class AcmeMap extends HTMLElement {
  static get extensionPoints() {
    return { edit: { shape: 'shapes/map-fields.ttl' } };
  }
  get source() { return this.getAttribute('source'); }   // where it reads config
  reload() { /* re-read source, re-render */ }            // called after a save
}

That's the whole opt-in — no glue. The same idea is open-ended: a capability can define other points (e.g. annotate), and any component listing that point in extensionPoints gets enhanced.

Step 4 — (Optional) Provide a capability others switch on

If you want to offer a capability — something people pull in with data-extend-with="acme-thing" that enhances other people's components — add a capabilities block to your manifest (Step 2), and write a module that finds and enhances every component offering your point, now or whenever one mounts later:

import { observeExtensionPoint } from 'solid-web-components/core/extension-points.js';

observeExtensionPoint('annotate', (el, spec) => {
  // attach your enhancement to `el` using `spec`
});

You never import the components you enhance; they never import you. You meet in the DOM.
