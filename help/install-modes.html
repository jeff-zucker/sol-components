Invoking Components

Everything is ES modules. You need three things on the page: the components you want (custom-element tags), one <script> that loads them, and an importmap so bare specifiers like rdflib resolve. The loader (option 0) bundles all three into a single tag and injects the importmap for you, so an app can stay a thin shell that is unaware of the dependency stack. If you want full control, drive the modules and importmap yourself (option 1).

0. Simplest: one tag — the loader

dist/sol-loader.min.js is a parser-blocking <head> script. It injects the right importmap, then import()s each module you name — in dependency order — and fires swc:ready. In data-bundles list the module specifiers you want: sol-basic (the no-RDF UI family) and any component (sol-feed, sol-time, sol-pod, menu-from-rdf, …). In data-with add capabilities — auth (⇒ sol-login), sparql (⇒ sol-query), rdf (the solid-ui editing stack ⇒ sol-tree-edit / sol-form / sol-settings), and solidos (⇒ mashlib + sol-solidos). 
<script src="../dist/sol-loader.min.js"
        data-stage="local"
        data-bundles="sol-basic sol-time sol-weather sol-feed"
        data-with="auth sparql rdf"></script>
Run code once everything is ready: 
<script>
  await SolidWebComponents.ready;        // or: document.addEventListener('swc:ready', …)
  // …or load programmatically:
  await SolidWebComponents.load(['sol-basic', 'sol-time'], { with: ['auth', 'rdf'] });
</script>
Deps ride in transitively. You never list rdflib, dompurify, or marked — the components import them, and the injected importmap resolves (and dedupes) them to one shared instance. That single-instance guarantee is what keeps term/store identity coherent across every module, so auth propagation and the editing stack just work.

data-stage — where deps come from

local resolves deps to the vendored copies in dist/vendor/*.js (offline-ready); cdn resolves them to esm.sh. Need a custom deployment? Point data-importmap="path/to/map.json" at your own importmap — that is the only time an app has to know the dep list. A page that already carries its own <script type="importmap"> is left untouched.

1. Importmap + ES modules (full control)

Skip the loader and drive the modules yourself. The browser needs an importmap so bare specifiers resolve; then import the component modules. The build already writes ready-made importmaps — dist/importmap-local.json (vendored) and dist/importmap-cdn.json (esm.sh) — each with one entry per dep and per component, so you can reference both bare rdflib and bare sol-feed.

Inline the map (works in every browser)

List only the deps your components actually use. For sol-include (dompurify + marked):

<script type="importmap">
{
  "imports": {
    "dompurify": "../dist/vendor/dompurify.js",
    "marked":    "../dist/vendor/marked.js"
  }
}
</script>
<script type="module" src="../web/sol-include.js"></script>

The editing stack (sol-form / sol-settings / sol-tree-edit)

These read & write data as RDF, so they need solid-logic + solid-ui. Load them in order — solid-logic before solid-ui (which sets window.UI) before the components:

<script type="importmap">
{
  "imports": {
    "rdflib":             "../dist/vendor/rdflib.js",
    "n3":                 "../dist/vendor/n3.js",
    "dompurify":          "../dist/vendor/dompurify.js",
    "marked":             "../dist/vendor/marked.js",
    "solid-logic":        "../dist/vendor/solid-logic.js",
    "solid-ui":           "../dist/vendor/solid-ui.js",
    "rdf-validate-shacl": "../dist/vendor/rdf-validate-shacl.js"
  }
}
</script>
<script type="module">
  import 'solid-logic';   // solid-logic BEFORE solid-ui
  import 'solid-ui';      // sets window.UI
  import '../web/sol-form.js';
</script>

<sol-form shape="shapes/x.shacl" subject="data/x.ttl#It"></sol-form>
Why inline the JSON? External-file importmaps (<script type="importmap" src="…">) work only in Chrome 124+, Firefox 132+, Safari 16.4+. Inline JSON works everywhere. The loader (option 0) inlines it for you. For cdn instead of vendored files, swap the ../dist/vendor/*.js paths for the https://esm.sh/… entries in importmap-cdn.json.

2. With a bundler (Vite, webpack, Rollup, esbuild…)

Building an app with a bundler? Install the package and import what you want. The bundler reads solid-web-components's exports map and resolves the right entry per runtime — browser builds get the web component, Node builds get the Node API.

npm install solid-web-components
import 'solid-web-components/web/sol-menu.js';
import 'solid-web-components/web/sol-include.js';
import 'solid-web-components/web/sol-feed.js';
// or all at once:
import 'solid-web-components/web';   // → web/sol-full.js

Heavy environment-specific deps (@comunica/query-sparql, the inrupt auth packages, solid-ui) are optional peer dependencies — your install only pulls what you ask for. Add them as needed:

npm install @comunica/query-sparql       # only if you query remote SPARQL endpoints

The capabilities live in the fixed dist/swc.manifest.json (identical for every app — you never edit it). Auth fetch propagates page-wide: once <sol-login> is present and the user is logged in, every <sol-query> / <sol-include> automatically uses the authenticated fetch — no wiring.

Troubleshooting

"Custom element 'sol-query' is not defined"

The module hasn't loaded yet. With the loader, await SolidWebComponents.ready (or the swc:ready event) before code that touches the element, and confirm the token is spelled like the module (sol-query, not query). In importmap mode, confirm the <script type="module"> URL resolves (open it — you should see JS, not a 404).

"Failed to resolve module specifier 'rdflib'"

  1. The importmap is missing the entry — add it, or use the loader, which injects a complete map.
  2. The importmap is loaded after the first <script type="module"> — order matters: 
    <script type="importmap">{ ... }</script>   <!-- always first -->
<script type="module"    src="..."></script>
  3. You used the external-file form in an older browser — inline the JSON.

404 on a vendored file (offline / data-stage="local")

Either npm run build:vendor hasn't run, or the importmap's relative path doesn't match where dist/vendor/ sits relative to the page. Importmap paths resolve against the importmap's own URL, not the page URL — the loader handles this with a __BASE__ placeholder pinned to its own location.
