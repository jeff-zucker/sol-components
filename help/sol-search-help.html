sol-search component

A multi-engine search field with two layouts: an icon-trigger that pops up a panel (view="button"), or an inline strip with the input and engine radios always visible (view="inline"). Engines come from a default list, a JSON attribute, or an RDF source — the same bookmark vocabulary <sol-feed> uses. Submitting opens the result in a shared reader window so repeat searches don't spawn new tabs.

<script src="dist/sol-full.umd.min.js"></script>

<sol-search default-engine="ddg"></sol-search>
<sol-search
  view="inline"
  default-engine="ddg"
></sol-search>
AttributeRequiredDescription
view optional button (default) renders an icon trigger that opens a panel on click; inline renders the field, Go button, and engines row directly with no toggle. Best inline use: a dedicated search strip; best popup use: a header where space is tight.
source optional file.ttl#TopicName — an RDF engine list (bookmark or SKOS vocabulary). See Engine sources.
engines optional JSON array of {id, label, url} — url is the search-URL prefix that receives the query as its query-string tail.
default-engine optional The id (or full URL) of the engine that starts selected. Falls back to the first engine.
placeholder optional Input placeholder text (default Search…).

Theming via CSS parts

Form, input, button, panel, and trigger are exposed as shadow parts: form, input, submit, trigger, panel.

sol-search::part(input)  { padding: .6rem .9rem; }
sol-search::part(submit) { background: var(--accent); }

Where the engine list comes from

Sources are checked in order — the first that yields a non-empty list wins:

1. source (RDF)

A Turtle file with a topic and a set of ui:Link entries. The bookmark vocabulary is the simplest:

@prefix ui: <http://www.w3.org/ns/ui#> .
@prefix bk: <http://www.w3.org/2002/01/bookmark#> .

<#SearchEngines> a bk:Topic ; ui:label "Search engines" .

:ddg  a ui:Link ; ui:label "DuckDuckGo" ;
      bk:recalls <https://duckduckgo.com/?q=> ;
      bk:hasTopic <#SearchEngines> .
:wp   a ui:Link ; ui:label "Wikipedia"  ;
      bk:recalls <https://en.wikipedia.org/w/index.php?search=> ;
      bk:hasTopic <#SearchEngines> .

Then point at the fragment:

<sol-search source="data/search-engines.ttl#SearchEngines"></sol-search>

SKOS works just as well (see sol-feed help — sol-search shares the same source-list parser).

2. engines (JSON attribute)

One-off override without RDF. Each item is { id, label, url }; the user's query is appended to url:

<sol-search view="inline" engines='[
  { "id":"ddg", "label":"DuckDuckGo", "url":"https://duckduckgo.com/?q=" },
  { "id":"g",   "label":"Google",     "url":"https://www.google.com/search?q=" }
]'></sol-search>

3. Built-in defaults

If neither source nor engines is given (or both fail), sol-search falls back to: DuckDuckGo, Google, Wikipedia, prefix.cc, LOV, Etymology, YouTube, Wayback. Set default-engine to one of those ids to pre-select it.

Layout: engines wrap to multiple rows

When the engine list is too wide for the container, the radios wrap onto a second (or third) row beneath the input/Go row. There's no carousel — just normal flex-wrap; centred via justify-content: center so each row reads as a centred cluster.

Accessibility

  • The input is a real <input type="search">; the Go button is a real <button type="submit">. Pressing Enter in the field submits the form — same as clicking Go.
  • view="button": the trigger is a labelled button with aria-haspopup="dialog" and aria-expanded. The panel is a role="dialog"; Escape closes it, and an outside-click closes it via document-level pointer-down listener (cleaned up on disconnect).
  • Engines are real <input type="radio"> controls inside <label> elements — tab into the group, arrow-keys to switch.
  • The first engine in the list is auto-selected on render so screen readers can announce a default choice.
  • All colours come from shared theme tokens, so the widget follows the page's light / dark setting.
