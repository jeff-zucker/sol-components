Common attributes

Many of these components share the same attributes, so once you learn them you know them everywhere. This page is the shared vocabulary — each component's own help page lists only its extras. All of these go right on the HTML tag, the same way you'd write class or href.

Quick reference

AttributeWhat it doesUsed by
source What the component loads — a file, a web address, or (for the data-driven components) a settings file written in Turtle, like data/settings.ttl#Settings. nearly all
region Where opened content appears: a CSS selector for an area of your page (e.g. #main), or a pop-up keyword (modal, floating, tab, window). sol-button, sol-menu, sol-query, sol-accordion, sol-weather
handler Which component opens the content. Usually optional — a sensible default is chosen. Set it (e.g. sol-query) to force a particular one. sol-button, sol-menu, sol-tabs, sol-modal, sol-pod
from-rdf Build the component's items from a Turtle file instead of writing them out in HTML. sol-menu, sol-tabs, sol-dropdown-button, sol-settings
view Pick a display style. The available choices depend on the component (each page lists its own). sol-feed, sol-search, sol-query, sol-calendar, sol-form, sol-live-edit, sol-pod-ops
login Point a component at a specific <sol-login> (a CSS selector) when a page has more than one. Otherwise it finds the login on the page automatically. sol-include, sol-query, sol-pod, sol-pod-ops, sol-login
if-logged-in React to whether the user is signed in. On sol-menu it's a yes/no flag (show this item only when logged in); on sol-include it's an alternate source to show once logged in. sol-menu, sol-include
proxy A CORS-proxy web address put in front of cross-site fetches, so a component can read a resource that wouldn't otherwise allow it. sol-feed, sol-calendar (or set once on <sol-default>)
trusted Let same-site HTML render as real HTML. Without it, fetched HTML is shown as plain text. Use only for content you control. sol-include, sol-button, sol-modal
raw Show a file's contents as-is (its literal text) instead of interpreting it. sol-include
selector Show only part of a fetched document — give a CSS selector and just that piece is pulled in. sol-include
orientation Lay the items out horizontal or vertical. sol-menu, sol-tabs, sol-dropdown-button
editable Allow editing of what's shown, right in place. sol-feed, sol-form
editor-self Add a small gear so the component can edit its own settings on the page. sol-calendar, sol-menu, sol-search, sol-time, sol-weather, sol-tabs
shape A SHACL file describing the fields of an edit form (which boxes to show and what type each is). sol-form (and any component you make editable)
save-to Where edits are written back. sol-form

A closer look

source — what to load

The one attribute almost every component takes. For the content components it's a file or web address; for the data-driven ones it points at a small Turtle settings file, usually with a #Name on the end to pick the right entry inside it:

<sol-include source="readme.md"></sol-include>
<sol-time source="data/time-settings.ttl#Settings"></sol-time>

region & handler — what opens, and where

The components that launch things (buttons, menu items, tabs) use this pair: handler says what opens the content, region says where it lands.

<!-- open settings.html in the #main area -->
<sol-button source="settings.html" region="#main">Settings</sol-button>

<!-- pop a query up in a dialog -->
<sol-button handler="sol-query" source="data/reports.ttl" region="modal">Reports</sol-button>

You don't have to repeat region on every launcher. Put it once on a wrapping element with data-region="…", or on <sol-default> (below), and the launchers inside use it.

from-rdf — items from a data file

Menus, tab strips, and dropdowns can be written out in HTML, or generated from a Turtle file so the same data can drive several places at once:

<sol-menu from-rdf="data/menu.ttl#MainMenu" region="#main"></sol-menu>

login & if-logged-in — being sign-in aware

Drop a <sol-login> on the page and the data components use the signed-in user's access automatically — no wiring. The two attributes here only matter when you want finer control:

<!-- show this menu item only to signed-in users -->
<a href="data/private.ttl" if-logged-in>My private data</a>

<!-- show one page to guests, another once signed in -->
<sol-include source="guest.html" if-logged-in="owner.html"></sol-include>

<!-- with two logins on the page, target one explicitly -->
<sol-query login="#alt" source="data.ttl"></sol-query>

Set shared options once — <sol-default>

Rather than repeating the same setting on every component, put a single <sol-default> on the page. A component checks its own attribute first and falls back to <sol-default> for shared options like the CORS proxy, the default region, and the default sign-in provider:

<sol-default
    proxy="https://example.org/proxy?uri="
    region="#main"
></sol-default>

Change a value on <sol-default> at runtime and the components that rely on it update to match.
