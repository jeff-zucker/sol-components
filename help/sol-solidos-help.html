sol-solidos

A custom element that mounts the SolidOS data browser inside its light DOM. Setting the source attribute drives outliner.GotoSubject(...) so the panes navigate to that resource.

Attributes

  • source — URL of the Solid resource to display. Setting it after the element is connected triggers a navigation. If unset on first mount, the panes initialise with window.location.href as the subject.

Runtime dependencies

sol-solidos relies on mashlib + solid-panes (a substantial dependency tree — RDF store, panes, login UI, theme module). mashlib is a heavy peer that sol-solidos does not bundle in. The loader's data-with="solidos" capability pulls it in for you (rdflib + mashlib), in order, and injects the importmap — so window.panes, window.SolidLogic, and window.Mashlib are populated before sol-solidos's connectedCallback runs. You don't name the deps yourself.

<script src="../dist/sol-loader.min.js"
        data-stage="local"
        data-with="solidos"></script>

<sol-solidos source="https://your.pod/some/resource"></sol-solidos>

Known mashlib quirk: empty window.Mashlib

mashlib's UMD wrapper sets window.Mashlib to its entry-point exports — and its entry point (./src/index.ts) is pure side-effects with no exports. So window.Mashlib ends up an empty {}. sol-solidos's getMashlib() looks for Mashlib.initMainPage, which is actually exposed on window.panes. Patch it before mounting sol-solidos:

<script type="module">
  await SolidWebComponents.ready;   // mashlib globals are populated by now
  if (window.Mashlib && window.panes && window.panes.initMainPage) {
    window.Mashlib.initMainPage = window.panes.initMainPage;
  }
</script>
<sol-solidos></sol-solidos>

Header / footer hidden by default

sol-solidos CSS-hides its own <header id="PageHeader"> and <footer id="PageFooter">, because when sol-solidos is mounted inside a host shell (data-kitchen, …) the host already owns login + help/prefs chrome and a duplicate login button in mashlib's header is broken anyway (popup-mode pod sessions don't reach mashlib's default Inrupt session, so its login button stalls). Mashlib's reads still go through rdflib's patched Fetcher → solFetch → the host's <sol-login> chip handles any sol-auth-needed prompt and retries.

If you want mashlib's standalone header back, override the rule in your page CSS:

sol-solidos > #PageHeader,
sol-solidos > #PageFooter { display: block !important; }

Embedding alternative: skip sol-solidos, use panes directly

sol-solidos creates its own DOM scaffolding (<header id="PageHeader">, <main id="mainContent">, <footer id="PageFooter">) and then calls initMainPage, which creates a parallel set of SolidOS-shell elements (#mainSolidUiHeader, #MainContent, #PageFooter) at the document body. If you want the SolidOS chrome (banner + side menu + footer) and the outline without the duplicate IDs, skip sol-solidos and use mashlib's lower-level API directly:

<body id="PageBody">
  <solid-ui-header id="mainSolidUiHeader"><span slot="title"></span></solid-ui-header>
  <main id="MainContent" role="main">
    <div class="app-shell">
      <aside id="NavMenu" class="app-nav" hidden><div id="NavMenuContent"></div></aside>
      <div class="app-view"><table id="OutlineView" class="outline-view"></table></div>
    </div>
  </main>
  <footer id="PageFooter"></footer>
  <script type="module">
    await SolidWebComponents.ready;   // window.panes / window.SolidLogic ready
    const store = window.SolidLogic.solidLogicSingleton.store;
    window.panes.initMainPage(store, 'https://your.pod/foo');
  </script>
</body>

This is the structure mashlib's databrowser.html uses; initMainPage fills the existing elements instead of creating new ones. For navigation after the initial render, hold the outliner and call it directly:

const $rdf = await import('rdflib');   // bare specifier resolved by the loader's importmap
const outliner = window.panes.getOutliner(document);
outliner.GotoSubject($rdf.sym(newUrl), true, undefined, true, undefined);
