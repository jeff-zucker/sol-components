Loading the components

The loader is one <script> tag you add near the top of your page. It reads a manifest that describes the components, makes them available, and then you can use tags like <sol-button> or <sol-menu> anywhere in your HTML.

The loader tag

Add it once, in the <head> of your page:

<script src="https://cdn.jsdelivr.net/npm/component-interop@0.2.0/component-interop.js"
        data-stage="local"
        data-manifest="../dist/sol-components.manifest.json"
        data-components="sol-basic sol-feed"
></script>

It has three knobs:

  • data-manifest — the manifest that describes the components. The loader knows nothing about any one library on its own; this is what points it at sol-components (list several manifests to mix libraries).
  • data-components — the components (or bundles) you want; each token is a tag name like sol-feed, or a bundle name like sol-basic for the everyday UI. Use * for everything the manifest offers.
  • data-stage — where the code is loaded from: local for files you host yourself, cdn for loading over the internet, or auto to pick local on localhost and cdn elsewhere.

You don't switch capabilities on by hand any more. A data capability like editing loads automatically when the attribute that needs it appears on the page — for example, data-edit-shape pulls in the editing stack, and data-from-query pulls in the query engine. To guarantee a capability is present up front, just name its component in data-components (see the table below).

Run your code when it's ready

If you have JavaScript that uses the components, wait for them to be ready first:

<script>
  await SolidWebComponents.ready;
  // your code here
</script>

If you'd rather listen for an event, the loader fires swc:ready on the document when everything is loaded.
