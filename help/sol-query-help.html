sol-query

Pulls data out of a data file or a public data service and shows it on your page — as a table, a searchable list, a rolodex, and more. You say where the data is and (optionally) what to ask for; the component fetches it and displays the results. If the page also has a <sol-login> and the data is private, it uses the signed-in user's access to read it.

Try it

  • view="rolodex"
  • view="auto-complete"
  • view="table"
  • view="accordion"
  • view="dl" (description list)

Use it in your HTML

Add this line once, near the top of your page — it makes the component available:

<script src="dist/sol-loader.min.js" data-bundles="sol-query" data-extend-with="auth sparql"></script>

Then point the tag at some data and pick how to show it:

<sol-query endpoint="https://example.org/data.ttl" view="table"></sol-query>

Options

Some attributes here are shared across components — see Common attributes.

AttributeWhat it does
endpoint Where the data is — the address of a data file or a public data service. You can list more than one, separated by commas, to pull from several at once. Required.
sparql The question to ask, for finer control than just "everything." Either a SPARQL query written out inline, or the address of one you've saved in a query file.
pattern A simpler way to ask a question: a single triple pattern such as ?person schema:name ?name. Use this instead of sparql when you just want matching items.
var-… Fill in a blank in a saved query. A saved query can contain a placeholder like {{limit}}; set it from the tag with var-limit="2".
view How to display the results: table, dl (a description list), rolodex, accordion, or auto-complete (a search box). You can also give the address of a custom view. This is a shared attribute — see Common attributes.
placeholder The greyed-out hint text inside the search box, for the auto-complete view.
go-label The label on the search button, for the auto-complete view.
callback The name of a function on your page to run when the user picks a result in the auto-complete view. It receives the chosen item's address and label.

For a saved query, the query text is expected to live in the query file under the rdfs:comment predicate. There's a ready-made shape for query libraries you can follow.

Examples

<!-- show everything in one data file, as a table -->
<sol-query endpoint="https://example.org/data.ttl" view="table"></sol-query>
<!-- ask a specific question with an inline query -->
<sol-query

    endpoint="https://example.org/data.ttl"
    sparql="SELECT ?s ?p ?o WHERE {?s ?p ?o.}"
    view="dl"

></sol-query>
<!-- find matching items with a simple pattern -->
<sol-query

    endpoint="https://example.org/data.ttl"
    pattern="?person schema:name ?name"

></sol-query>
<!-- pull from two places at once, using a saved query -->
<sol-query

    endpoint="https://example.org/endpoint1, https://example.org/endpoint2"
    sparql="https://example.org/myQueryLibrary.ttl#MyQuery"

></sol-query>
<!-- fill in a blank in a saved query -->
<sol-query

    endpoint="data/sample-data.ttl"
    sparql="data/myQueryLibrary.ttl#MyReplaceableQuery"
    var-limit="2"

></sol-query>

If the data you point at is private, put a <sol-login> on the page too. Once the user signs in, sol-query reads their private data automatically — no extra setup.
