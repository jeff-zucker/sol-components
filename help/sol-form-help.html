Shape-driven vs form-driven

sol-form has two rendering paths. The choice depends on whether you have a SHACL shape (use shape-driven) or a hand-authored ui:Form TTL (form-driven). They produce identical edits to the data file.

Shape-driven (recommended for new components)

<sol-form
  shape="shapes/weather-settings.shacl"
  subject="data/weather-settings.ttl#Settings">
</sol-form>

sol-form fetches the SHACL shape, calls parseShape() in core/shape-to-form.js to extract a normalized property descriptor list, and renders one input per property. Cardinality, datatype, enum, and nodeKind constraints all come from the shape — no separate form file needed.

The SHACL shape is also handed to rdf-validate-shacl before every save. Edits that violate the shape surface as validation errors and the PUT is skipped.

Form-driven (legacy)

<sol-form
  source="data/menu-form.ttl"
  subject="data/menu.ttl#MainMenu">
</sol-form>

sol-form fetches the ui:Form definition and hands it to solid-ui's form widgets. The form file is hand-authored TTL using the http://www.w3.org/ns/ui# vocabulary (ui:SingleLineTextField, ui:Multiple, ui:Choice + ui:from, etc.). Still used by sol-tabs's editor (sol-menu's editor moved to the shape-driven sol-tree-edit over menu.shacl).

If you need a persistent form-as-an-artifact in shape-driven mode (for docs, code review, or non-solid-ui tooling), use the CLI to materialise one from any shape file.

SHACL constructs shape2form recognises

The list below covers the shape-driven path. The runtime renderer in core/shape-to-form.js walks each sh:property entry on the NodeShape and matches against this table.

SHACLRenders asNotes
sh:targetSubjectsOf—Finds the subject(s) to edit by structural match. sh:targetNode <#X> resolves relative URIs against the shape's URL (not the data's), so it almost never matches what you mean — use sh:targetSubjectsOf instead.
sh:property + sh:pathOne row in the formEach sh:property describes one editable predicate. The sh:path is the real RDF predicate the input binds to (e.g. geo:lat, schema:timezone, dct:source).
sh:datatype xsd:stringtext inputSingle-line text input.
sh:datatype xsd:integernumber inputWhole-number input.
sh:datatype xsd:decimalnumber inputWith step="any" for fractional values.
sh:datatype xsd:booleancheckboxStored as "true"/"false" typed literal.
sh:datatype xsd:anyURIurl inputStored as typed literal. Use sh:nodeKind sh:IRI instead to store as a NamedNode.
sh:in (literals)<select> dropdownEnumerated allowed values; the shape's list becomes the options.
sh:in (IRIs)<select> dropdownFor NamedNode enums (e.g. qudt:SI), the select stores the URI but displays an rdfs:label from the shape — author labels next to the URIs (see shapes/weather-settings.shacl for the pattern).
sh:nodeKind sh:IRIurl inputValue stored as a NamedNode (<url> in turtle), not a literal.
sh:minCount 1Required (placeholder seeded)If the data has no value for the predicate, a blank one is added on render so the user has somewhere to type.
sh:maxCount > 1 or absentRepeat controlsThe field renders with + Add another and − buttons for each value.
sh:nameLabel textDisplayed next to the input.
sh:descriptionTooltipSet as the label's title attribute.

Constructs not currently recognised (treated as plain text inputs): sh:pattern, sh:minInclusive, sh:maxInclusive, sh:minLength, sh:maxLength. sh:or and sh:and are in the next batch — see claude/plans/PLAN-shape2form-or-and.md. Adding more is additive: new branches in core/shape-to-form.js's readShapeProperty and buildShapeInput.

Previously: PropertyValue indirection (now removed)

Earlier versions of shape2form recognised the schema:additionalProperty + sh:qualifiedValueShape pattern (every setting reified as a schema:PropertyValue with schema:name + schema:value). That pattern was migrated out in favour of direct predicates — see claude/plans/PLAN-vocab-migration.md. Component settings now bind to standard W3C / Schema.org / Dublin Core / QUDT / OWL-Time predicates directly.

The one remaining use of PropertyValue indirection in the codebase is menu items' ui:attribute Multiple — that's deliberate, because menu Component items carry HTML attributes whose names depend on the element being instantiated (an open schema, where indirection is the right tool).

shape2form CLI

A Node script that emits a static ui:Form TTL from a SHACL shape — the persistent "third artifact" mentioned in the Modes tab. Useful for documentation, code review, and consumption by tooling that expects a ui:Form file (other than solid-ui's vanilla renderer — see caveat below).

Usage

node scripts/shape2form.mjs <input.shacl> [--out <output.ttl>]
node scripts/shape2form.mjs <input.shacl> --stdout
node scripts/shape2form.mjs <input.shacl> --name MyForm

Example

Running it on the weather shape:

$ node scripts/shape2form.mjs shapes/weather-settings.shacl --stdout

After the direct-predicate migration

After the vocab migration (see claude/plans/PLAN-vocab-migration.md), the generated forms point ui:property at the actual editable predicate — geo:lat, dct:source, schema:timezone, etc. — instead of the old schema:additionalProperty placeholder. The generated forms are now solid-ui-renderable for the first time: drop one into <sol-form source="…this-file.ttl" subject="…data.ttl#X"> for the legacy form-driven path, or use <sol-form shape="…" subject="…"> for the same fields rendered via shape-to-form's runtime.
