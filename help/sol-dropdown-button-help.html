sol-dropdown-button component

A trigger button that drops a menu in a floating popup. Declare the items inline in a <menu>: each item's handler says what it does — a bare name is a command (it fires an event for your app to handle on click), a custom-element tag or an <a href> mounts a component. Nothing is pre-selected; the popup opens on click and closes on pick / outside-click / Escape.

Authoring

The items live in a single <menu> child. An item whose handler is a bare name (no hyphen, not a registered element) is a command: clicking it dispatches sol-command with detail: { command, params } — your page listens and acts. An item that is an <a href>, or whose handler is a component tag, mounts that content instead; point it at a region (e.g. region="modal") to choose where it surfaces. Nest a <submenu> for a fly-out group.

Mark an item requires-write (or if-logged-in) and it is exposed as ::part(requires-write) so the host page can hide or disable it for guests — the dropdown itself takes no policy.

Attributes

AttributePurpose
label Trigger text (default ⋮).
handler (on an item) Bare name → command (fires sol-command); component tag → mount that element.
params (on a command item) JSON forwarded in the sol-command detail.
requires-write (on an item) Flag an item as owner-only; surfaced via ::part(requires-write).

The trigger is exposed as ::part(trigger).

Examples

<!-- command items: each fires sol-command on click -->
<sol-dropdown-button label="⋮ More">
  <menu>
    <button handler="install" requires-write>Install on my Pod…</button>
    <button handler="export" params='{"format":"csv"}'>Export</button>
    <a href="about.html" region="modal">About</a>
  </menu>
</sol-dropdown-button>

<script>
  document.addEventListener('sol-command', (e) => {
    const { command, params } = e.detail;
    // dispatch to your app's handlers…
  });
</script>
