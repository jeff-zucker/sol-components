Add your component to the ecosystem

Newly implemented. The surface this guide describes — window.SolidWebComponents.services / .rdf / .fetch / .defaults / .EVENTS, the loader's data-manifest / data-importmap-extra / registerCapability, and static get extensionPoints() — is now shipped. It hasn't yet been exercised end-to-end in a real browser, so treat the runtime details as provisional and report anything that doesn't behave as written.

You wrote a web component. Here's how to make it a good citizen of a solid-web-components page — so it loads alongside everyone else's components, shares the same data and login, and can be enhanced by capabilities like editing without you wiring anything up.

There are three things you can do, each optional and independent:

  1. Plug in — so it loads and coexists.
  2. Share resources — use the page's one data store, login, and settings.
  3. Offer extension points — let capabilities enhance your component.

You never import any solid-web-components source code. Everything is plain custom elements, the window.SolidWebComponents object, and DOM events.

1. Plug in (today)

Your component is a standard custom element. Two conventions make it a good neighbour. Pick a hyphenated tag with your own prefix so it never collides:

customElements.define('acme-map', AcmeMap);

Emit events that bubble and compose, named your-tag-action, so anything on the page can react without knowing about you:

this.dispatchEvent(new CustomEvent('acme-map-select', {
  bubbles: true, composed: true, detail: { id }
}));

Put it on the page. A page using the loader just names your module in data-bundles; data-manifest tells the loader where to find it (see step 4):

<script src="dist/sol-loader.min.js"
        data-bundles="sol-basic acme-map"
        data-manifest="https://acme.example/swc-manifest.json"
></script>

Or, simplest of all, the page author adds their own <script type="module"> and writes <acme-map> — no manifest needed.

Sharing libraries: if your component uses rdflib, import it as a bare specifier (import { sym } from 'rdflib') and mark it external in your build. The page already provides one shared copy through its importmap — that single copy is what keeps every component's data identical. Don't bundle your own.

2. Share resources

Everything shared lives on one object the loader puts on the page: window.SolidWebComponents. Wait for it to be ready, then use it:

const SWC = window.SolidWebComponents;
await SWC.ready;            // components + capabilities have loaded

What's on it

You want…Use
The one shared data storeSWC.rdf.store, await SWC.rdf.load(url)
The signed-in fetch (carries the user's credentials when logged in)await SWC.fetch(url)
React to sign-in / sign-outSWC.on(SWC.EVENTS.LOGIN, fn) / SWC.on(SWC.EVENTS.LOGOUT, fn)

So a component that shows a "save to my pod" button only when login is available, and re-renders when the user signs in:

if (SWC.has('auth')) this.showSaveButton();
SWC.on(SWC.EVENTS.LOGIN,  () => this.reload());
SWC.on(SWC.EVENTS.LOGOUT, () => this.reload());
const res = await SWC.fetch('https://my.pod/private.ttl');

No imports — your component and every sol-* component read and write the same store and use the same login.

3. Offer extension points

This is the interesting part: you can let capabilities improve your component without importing anything or knowing which capability will show up.

Declare, on your class, the extension points you offer. The most useful one is edit: declare it with a SHACL shape and the editing capability (loaded with data-extend-with="rdf") gives your component a settings form and a ⚙ gear, for free:

class AcmeMap extends HTMLElement {
  static get extensionPoints() {
    return {
      edit: { shape: 'https://acme.example/shapes/map.shacl' },
    };
  }
  get source() { return this.getAttribute('source'); }  // where it reads config
  reload() { /* re-read source, re-render */ }           // called after a save
}

That's the whole opt-in. When a page includes both your component and the rdf capability, <sol-settings> discovers acme-map, builds an editor from your shape, and calls your reload() after a save — and you wrote no glue.

The mechanism is open-ended: a capability can define new points (e.g. annotate, translate), and any component that lists that point in extensionPoints gets enhanced. You only declare the points you want. static get shape() / static get editor() (the form swc's own components use today) are just shorthand for the edit point.

4. (Optional) Ship a capability others extend with

If you want to provide a capability — something pulled in by data-extend-with="acme-thing" that enhances other people's components — publish two things.

A manifest fragment (what data-manifest points at), listing your capability's modules:

{
  "capabilities": {
    "acme-thing": { "modules": ["acme-enhancer"] }
  }
}

And a capability module that finds and enhances every component offering your point — now, or whenever one is added later:

import { observeExtensionPoint } from 'solid-web-components/core/extension-points.js';

observeExtensionPoint('annotate', (el, spec) => {
  // attach your enhancement to `el` using `spec`
});

You never import the components you enhance; they never import you. You meet in the DOM.
