sol-calendar component

An inline calendar viewer. Point it at any public iCalendar (ICS) feed — Google Calendar, Apple iCloud, Outlook, Proton Calendar, or a Solid pod — and it renders the upcoming events as an agenda list that fits whatever container the page gives it. v1 ships the agenda view; mini (today-only card) and month (grid + day popover) are next.

<script src="dist/sol-calendar.umd.min.js"></script>

<sol-calendar
  provider="google"
  calendar-id="en.usa#holiday@group.v.calendar.google.com"
  window-days="60"
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>
AttributeRequiredDescription
source one of source or calendar-id An ICS URL, or several whitespace-separated ICS URLs (amalgamated agenda), or file.ttl#Subject pointing at a schema:PropertyValue config block (which may itself declare multiple "source" values). See RDF config and Multi-source.
provider optional google | apple | outlook | proton | ics (default ics). Drives the header label and, for google, lets you supply calendar-id instead of the full URL.
calendar-id required iff provider="google" without source The calendar's email/id. The component builds the public ICS URL itself.
view optional agenda (v1 default and only value). Other views are planned.
start optional ISO date YYYY-MM-DD. Defaults to today's local midnight so same-day events still appear.
window-days optional Agenda lookahead in days (default 30).
max-events optional Cap on rendered events (default 100).
proxy often required CORS proxy pattern; same shape as <sol-feed>'s.
time-zone optional IANA TZ override (default: the browser's resolved zone).
locale optional BCP-47 locale for day headers and times.
hide-header optional (boolean) When present, omits the title + provider strip above the agenda. Useful when the host page already labels the slot (dashboards, sidebars).

proxy attribute

None of Google, Apple, Outlook, or Proton serve their public ICS endpoints with permissive CORS, so a browser cannot fetch them directly. sol-calendar tries the bare URL first, then retries through the proxy pattern. If the pattern contains the literal token {url} it is replaced with the encoded URL; otherwise the encoded URL is appended to the end of the pattern.

Append style (the URL is added to the end): 
proxy="http://localhost:3002/proxy?uri="
Token style (the URL is substituted for {url}): 
proxy="https://proxy.example/fetch?u={url}"

Omit proxy only for endpoints that already send Access-Control-Allow-Origin — typically a Solid pod or a self-hosted ICS file.

Provider hint

The provider attribute is a small hint — it labels the header and, for Google, lets the component build the URL itself. Every provider's data arrives in the same ICS format; we never authenticate, so only publicly-shared calendars are reachable today.

provider="google"

Give either a full ICS URL or just the calendar id; the component builds https://calendar.google.com/calendar/ical/<id>/public/basic.ics when only the id is supplied.

<sol-calendar provider="google"
  calendar-id="en.usa#holiday@group.v.calendar.google.com"
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>

provider="apple"

In Apple Calendar, share a calendar publicly and copy the webcal:// or https:// link. Pass it as source.

<sol-calendar provider="apple"
  source="https://p01-caldav.icloud.com/published/2/..."
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>

provider="outlook"

In outlook.com, publish a calendar and copy the ICS link from the sharing settings.

<sol-calendar provider="outlook"
  source="https://outlook.live.com/owa/calendar/.../calendar.ics"
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>

provider="proton"

In Proton Calendar, share a calendar publicly and copy the "Subscribe with calendar app" link.

<sol-calendar provider="proton"
  source="https://calendar.proton.me/api/calendar/v1/url/.../calendar.ics"
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>

provider="ics" (default)

Any ICS URL — a Solid pod, a self-hosted file, any provider not listed above.

<sol-calendar source="https://example.org/team.ics"></sol-calendar>

Settings from RDF

Instead of inlining every attribute in HTML, point source at a Turtle file with a schema:PropertyValue settings block. Setting names map to the matching HTML attributes ("source", "provider", "calendar-id", "view", "window-days", "max-events", "proxy", "time-zone", "locale"). Any HTML attribute still wins, so a page can override one field inline while the rest come from the TTL.

<sol-calendar source="data/calendar.ttl#Settings"></sol-calendar>

And data/calendar.ttl:

@prefix schema: <http://schema.org/> .

<#Settings>
  schema:additionalProperty
    <#source>, <#provider>, <#view>,
    <#window-days>, <#proxy> .

<#source>       a schema:PropertyValue ; schema:name "source"@en       ;
  schema:value "https://calendar.google.com/calendar/ical/.../public/basic.ics" .
<#provider>     a schema:PropertyValue ; schema:name "provider"@en     ; schema:value "google" .
<#view>         a schema:PropertyValue ; schema:name "view"@en         ; schema:value "agenda" .
<#window-days>  a schema:PropertyValue ; schema:name "window-days"@en  ; schema:value 30 .
<#proxy>        a schema:PropertyValue ; schema:name "proxy"@en        ;
  schema:value "http://localhost:3002/proxy?uri=" .

The PropertyValue pattern keeps the file free of invented predicates — only standard schema.org terms (PropertyValue, name, value, additionalProperty) appear in predicate position. The schema.org namespace is always http://schema.org/ — never the HTTPS form.

This path needs rdflib on the page (it publishes window.$rdf), the same as <sol-feed>'s topic / all views.

Multi-source (amalgamated) agendas

One element can fetch several calendars in parallel and merge their events into a single sorted agenda. A flaky feed doesn't blank the rest — its error is reported in the status strip and the surviving feeds still render.

From HTML

Pass whitespace-separated URLs in source:

<sol-calendar
  source="https://example.org/team.ics
          https://www.w3.org/groups/cg/solid/calendar/export/"
  proxy="http://localhost:3002/proxy?uri="
></sol-calendar>

From RDF config

A single settings block can declare the "source" PropertyValue more than once (or once with multiple schema:value statements) — both forms are equivalent and yield the same multi-source behaviour:

<#All>
  schema:additionalProperty
    [ a schema:PropertyValue ; schema:name "source"@en ;
      schema:value "https://www.w3.org/groups/cg/solid/calendar/export/" ],
    [ a schema:PropertyValue ; schema:name "source"@en ;
      schema:value "https://www.w3.org/groups/wg/lws/calendar/export/" ],
    [ a schema:PropertyValue ; schema:name "source"@en ;
      schema:value "https://calendar.google.com/calendar/ical/<id>/public/basic.ics" ],
    [ a schema:PropertyValue ; schema:name "window-days"@en ; schema:value 60 ] .

Then point the element at the fragment:

<sol-calendar source="data/calendar.ttl#All" hide-header></sol-calendar>

Failure handling

Each feed is fetched independently via Promise.allSettled. When one or more fail, the live status region shows "Loaded N of M calendars — K failed" and each failure is logged to the console with its URL. Successful feeds still render.

Every merged event carries a short calendar label derived from its source URL (e.g. w3.org/solid, google) — available on the JS event object for callers that want a per-source visual indicator. The current renderer doesn't show it; future versions may.

Meeting links (clickable event titles)

Different calendar publishers put video-meeting URLs in different places. sol-calendar looks for one in priority order and, when found, uses it as the href on the event's summary — so clicking the title opens the meeting room directly.

  1. The standard ICS URL: property of the VEVENT.
  2. LOCATION:, if the whole field is a URL (common for Jitsi-only calendars like the Solid CG one). The location text-row is suppressed in this case so the same URL doesn't render twice.
  3. The first URL inside DESCRIPTION: that matches a known meeting host: meet.jit.si, zoom.us, meet.google.com, teams.microsoft.com, whereby.com.
  4. The first URL of any host inside DESCRIPTION: — falls back to a W3C events-page link, a Google Group link, etc., when no inline meeting URL is published.

Trailing sentence punctuation (., ,, ;, )) is trimmed off URLs caught from DESCRIPTION text. Mixed LOCATION fields (text plus a URL) stay as plain text — only fully-URL locations are consumed as the href, to avoid disappearing "Room 4" from a location like "Room 4, also https://meet.jit.si/foo".

Theming

sol-calendar reads from the shared design tokens in web/styles/root.css. Override any of these on :root (global) or on the host element (scoped) to restyle a particular instance.

Tokens used

TokenUsed for
--bg page background (inherited; the calendar's outer host has no own bg)
--surface background of the agenda card
--border card border and row separators
--text event summary text
--text-muted date column, time column, location text, status strip
--accent "today" row stripe; fallback for --link
--link colour of clickable event summaries (meeting links)
--font-ui, --font-size typography on the host

Scoped override example

/* Make this one calendar's links red */
sol-calendar.team { --link: #c0392b; }

Accessibility

  • The agenda list is a labelled region; each day's events are grouped under a <section> with the day header as a sticky heading.
  • Event URLs are real <a> elements that open in a new tab with rel="noopener noreferrer".
  • Loading and error messages are announced through a polite aria-live region.
  • Today's day-header is visually highlighted; the highlight uses the shared --accent token so it inherits the page's light / dark setting.
  • All colours come from shared theme tokens.
