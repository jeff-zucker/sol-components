sol-settings component

A zero-config settings page. Drop one <sol-settings> on a page and it discovers every editable sol-* widget already in the DOM and renders one accordion panel to edit each — no wiring, no per-widget configuration.

Usage

<sol-weather place="Portland, OR"></sol-weather>
<sol-menu from-rdf="data/menu.ttl#MainMenu" region="#main"></sol-menu>

<sol-settings></sol-settings>

On connect, sol-settings walks the document — crossing into every shadow root — for elements whose custom-element class declares an editor. Here it finds the weather card and the menu, and builds a panel for each. It takes no attributes.

How a component becomes editable

A component opts in by declaring, on its class, one of:

DeclarationEditor used
static get shape() → a SHACL URL sol-form in shape-driven mode (e.g. sol-weather, sol-time).
static get editor() → { tag, subjectAttr, attrs } that explicit editor element (e.g. sol-menu → sol-tree-edit over menu.shacl).
static get editor() → a ui:Form TTL URL sol-form in form-driven mode (legacy).
static get editor() → { inline: true } opts out — not shown in settings.

Each panel's editor is lazy-mounted on first expand, wired with the host widget's source / from-rdf as the subject being edited. The panel label comes from the widget's label attribute, else its tag name.

Saving

Edits save through the editor's normal path (sol-form PATCHes via kb.updater). When an embedded editor emits sol-form-save, sol-settings calls reload() on the corresponding live widget so the page reflects the change immediately.

Discovery timing

Discovery only sees widgets present in the DOM when it runs. Two notes:

  • Widgets kept in a keep-alive region pane (mounted but hidden) stay discoverable across navigation; widgets torn down on nav-away are not.
  • refresh() re-walks and rebuilds if the widget set changed — cheap no-op otherwise. It runs automatically on sol-tab-activate (so navigating to the settings tab picks up newly-mounted widgets); call it yourself after mounting an editable widget late.

See it in context in data-kitchen's Settings, which lists every chrome widget (menu, weather, clock, search, proxy defaults, …) from a single <sol-settings>.
