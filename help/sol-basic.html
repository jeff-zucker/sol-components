sol-basic components

The everyday UI components — they work from plain HTML and need no sign-in or data setup. Load them all with one line:

<script src="dist/sol-loader.min.js" data-bundles="sol-basic"></script>

This page is the terse, all-in-one list — each component below has a copy-paste example. For attributes many of them share (source, region, handler, orientation, if-logged-in, …) see Common attributes.

sol-button

A button that opens something (page, file, component, website) into a region or a pop-up. The label is the text between the tags.

AttributeWhat it does
source What to open — a page, file, or web address.
region Where it opens: a CSS selector, or modal / floating / tab / window.
handler Which component opens the content (usually optional).
name A short label for the content when it opens as a tab.
replace Reload the content fresh on every click.
inline Sit inline (e.g. as a toolbar control) rather than as a block.
raw Show the source's literal text instead of rendering it.
trusted Let same-site HTML render as real HTML.

One example using each attribute:

<sol-button source="settings.html" region="#main">Settings</sol-button>
<sol-button source="reports.ttl" region="modal" name="Reports">Reports</sol-button>
<sol-button source="report.html" handler="sol-query" replace>Run query</sol-button>
<sol-button source="help.html" inline>?</sol-button>
<sol-button source="notes.txt" raw trusted>Notes</sol-button>

sol-dropdown-button

A trigger button that drops a <menu> of choices. Each choice is a <button> (an action) or an <a href> (opens a page or component). Picking an action fires sol-command.

AttributeWhat it does
label Text on the trigger button (defaults to ⋮).
handler (on a choice) A plain word makes it an action; a component tag opens that component.
params (on a choice) Extra data to pass with an action, written as JSON.
requires-write (on a choice) Marks a choice owner-only, to hide/disable for visitors.

Write the button with a <menu> of choices inside it:

<sol-dropdown-button label="⋮ Actions">
  <menu>
    <button handler="rename">Rename</button>
    <button handler="export" params='{"format":"csv"}'>Export</button>
    <button handler="install" requires-write>Install on my Pod…</button>
    <a href="about.html" region="modal">About</a>
  </menu>
</sol-dropdown-button>

sol-include

Fetches a file (HTML, Markdown, text) and drops its contents into the page. Markdown is rendered for you.

AttributeWhat it does
source The file to fetch and show.
selector Pull in only the part matching this CSS selector.
raw Show the file's text exactly as written.
trusted Let the fetched HTML run as real HTML and activate nested components.
if-logged-in An alternate source to show once signed in.

One example using each attribute:

<sol-include source="header.html" trusted></sol-include>
<sol-include source="blog.html" selector="article:first-of-type"></sol-include>
<sol-include source="script.js" raw></sol-include>
<sol-include source="guest.html" if-logged-in="owner.html"></sol-include>

sol-menu

A navigation menu with a built-in content panel; the first link opens automatically. List <a href> links, and group them under a <submenu> with a <label> (nestable to any depth).

AttributeWhat it does
icon (on a link) Show an icon instead of text; the link text becomes its tooltip.
handler (on a link) Which component opens the link (defaults to showing the page).
if-logged-in (on a link) Show this link only to signed-in users.
orientation horizontal (top bar) or vertical (sidebar).
from-rdf Build the menu items from a Turtle file instead of HTML.

A horizontal menu whose links show off the per-link attributes:

<sol-menu orientation="horizontal">
  <a href="home.html" icon="home.svg">Home</a>
  <a href="data.ttl" handler="sol-query">Data</a>
  <a href="private.ttl" if-logged-in>My private data</a>
  <submenu>
    <label>Docs</label>
    <a href="quickstart.md">Quickstart</a>
  </submenu>
</sol-menu>

Or build the items from a Turtle file instead of writing them out:

<sol-menu from-rdf="data/menu.ttl#MainMenu"></sol-menu>

sol-tabs

A tabbed box; each link becomes a tab and its page loads into the panel below. With one tab the bar hides itself. A non-link child (e.g. a button) is placed on the tab bar as a toolbar control.

AttributeWhat it does
keep-alive Keep every tab's panel loaded so scroll and media survive a switch.
variant sub for the slimmer styling of nested tabs.
orientation horizontal or vertical.
handler Which component opens a tab (defaults to showing the page).
from-rdf Build the tabs from a Turtle file instead of HTML.

A vertical, kept-alive tab set; a nested set uses variant="sub":

<sol-tabs keep-alive orientation="vertical">
  <a href="overview.html">Overview</a>
  <a href="data.ttl" handler="sol-query">Data</a>
  <a href="about.html">About</a>
  <sol-tabs variant="sub">
    <a href="a.html">A</a>
    <a href="b.html">B</a>
  </sol-tabs>
</sol-tabs>

Or build the tabs from a Turtle file:

<sol-tabs from-rdf="data/tabs.ttl#Tabs"></sol-tabs>

sol-accordion

Collapsible sections, one open at a time, first open by default. Each child <div> holds two inner divs — the first becomes the header, the second the collapsible content.

AttributeWhat it does
start-closed Leave every panel collapsed on first render.

Each section is a <div> wrapping a header div and a content div; start-closed leaves them all collapsed at first:

<sol-accordion start-closed>
  <div><div>Section 1</div><div>Content 1</div></div>
  <div><div>Section 2</div><div>Content 2</div></div>
</sol-accordion>

sol-rolodex

A card-by-card browser with previous/next buttons and arrow-key navigation. Each child <div> is a card. It fires sol-select when the card changes, and has no attributes of its own.

Each card is a <div>:

<sol-rolodex>
  <div>Card 1 content</div>
  <div>Card 2 content</div>
</sol-rolodex>

sol-default

An invisible record of shared defaults; place one on the page and other components fall back to its attributes. Changing one fires sol-default-change and updates dependents live.

AttributeWhat it does
proxy A CORS-proxy address used as the shared default.
region The default region for launchers that don't set their own.

Put one on the page; other components fall back to these:

<sol-default region="#main" proxy="http://localhost:3002/proxy?uri="></sol-default>

Pop-up surfaces

The modal dialog and the floating window aren't tags you write yourself. You reach them through a launcher (a sol-button, a menu link, a dropdown choice) by setting its region:

  • region="modal" — a centered dialog with a dimmed backdrop.
  • region="floating" — a draggable, resizable window with no backdrop; several can coexist and the page behind stays interactive.

Set the region on the launcher:

<sol-button source="reports.ttl" region="modal">Reports</sol-button>
<sol-button source="notes.html" region="floating">Notes</sol-button>
